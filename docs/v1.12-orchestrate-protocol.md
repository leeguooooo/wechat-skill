# wechat-skill v1.12 — Orchestrate Protocol

**Audience**: 任何想让自家 SaaS 接入用户本机微信、但用户 Mac 没公网的开发者。

**Pattern**: outbox + webhook（Mac 全 outbound 调用，NAT-friendly）。

---

## 0. 角色 & 数据流

```
┌────────────────────────────────────┐
│ 你的 SaaS (CF Worker / VPS / etc.) │
│                                     │
│  - bot_outbox 表（你存）            │
│  - 4 个 endpoint（你实现）：         │
│    GET  /api/wechat-outbox/claim   │
│    POST /api/wechat-outbox/:id/done│
│    POST /api/wechat-outbox/:id/fail│
│    POST /api/wechat-inbound        │
│                                     │
│  - 订阅者各自一个 SaaS bearer token │
└────────────────────────────────────┘
                ▲ poll / push
                │ HTTPS + Bearer
                │
┌────────────────────────────────────┐
│ 订阅者 Mac (wechat-skill v1.12)     │
│                                     │
│  wechat orchestrate                 │
│    --outbox-url=<your saas base>   │
│    --webhook-url=<your saas inbound>│
│    --bearer=<token-per-tenant>     │
│                                     │
│  - poll outbox → 调本机 :18402 send│
│  - 订 SSE :18400 → POST webhook    │
└────────────────────────────────────┘
```

**关键不变量**：
- Mac 全部 outbound HTTP（NAT-friendly，订阅者不需要域名 / 公网 / cloudflared）
- SaaS 是真相源（state lives in your D1 / Postgres / KV，不存在 Mac 上）
- Bearer token 跟订阅者一对一，scope 只能操作该 tenant 的 outbox + inbound

## 1. 鉴权

每次 SaaS 给一个订阅者发激活/接入凭据时，签发**一次性 SaaS-side bearer token**：

```
HTTP Header:
  Authorization: Bearer <saas-issued-token>
```

**禁止**复用 wechat-skill 的 `wxp_tok_*` user_token —— 那是 wechat-skill SaaS 跟订阅者的关系，跟你的 SaaS 解耦。

Token scope 由 SaaS 自己定义。建议字段：

```json
{
  "tenant_id": "cherry-prod-customer-42",
  "scopes": ["outbox:claim", "outbox:complete", "inbound:write"],
  "iat": 1730000000,
  "exp": null
}
```

**TTL 建议**：长期（无 exp 或 ≥ 365d）。订阅者把 token 写进 `wechat orchestrate` 的 launchd plist env，频繁刷新会增加运维成本。

订阅者吊销由 SaaS 端做（revocation list / token rotation）。

## 2. Outbox 状态机

```
                  ┌─────────────────────┐
                  │       pending       │ ← SaaS 写入 (POST or trigger)
                  └─────────┬───────────┘
                            │ Mac orchestrate claim
                            ▼
                  ┌─────────────────────┐
                  │      claimed        │ ← lease_until = now + 60s
                  └────┬────────────┬───┘
              done    │            │   fail (retryable)     fail (hard)
                      │            │
          ┌───────────┘            └───────────┐                ┌───────┐
          ▼                                    ▼                ▼       ▼
   ┌─────────────┐                  ┌──────────────────┐  ┌─────────────┐
   │    done     │                  │ pending          │  │   failed    │
   └─────────────┘                  │ (next_attempt_at │  └─────────────┘
                                    │  = now + backoff)│
                                    └──────────────────┘

Lease expiration:
  claimed AND lease_until < now
  → automatically reset to pending (visibility timeout)
```

**lease_until** 是关键：Mac 崩了 / 网断了，超过 lease 后 SaaS 端自动把 row reset 成 pending，下一个 claim 拿到。

**默认值**：
- lease 60 秒
- backoff 序列：30s / 2min / 10min / 30min（每次 ×4）
- max retry：5 次 → 转 `failed`

## 3. Endpoints

### 3.1 `GET /api/wechat-outbox/claim?limit=N`

**用途**：Mac orchestrate 请求 N 条任务，SaaS 原子 claim 把它们标 `claimed` 返回。

**Request**:
```http
GET /api/wechat-outbox/claim?limit=10
Authorization: Bearer <saas-token>
```

**Response 200**:
```json
{
  "rows": [
    {
      "id": "out_abc123",
      "idempotency_key": "cherry-order-1234-msg-1",
      "to": "wxid_xxxx OR room_id@chatroom OR display_name",
      "text": "您好，订单已确认...",
      "claimed_at": "2026-04-28T10:00:00Z",
      "lease_until": "2026-04-28T10:01:00Z",
      "attempt": 1
    }
  ]
}
```

**字段说明**：
- `id`：SaaS 内部主键，Mac 透传，回 done/fail 用同一个 id
- `idempotency_key`：**必填**。SaaS 跨 retry 用同一个 key（`cherry-order-${id}-msg-${seq}`），防止 Mac 重复 send 时重复扣
- `to`：recipient hint（wxid / room_id / 备注名 / 群名）。Mac 调本机 `/v1/send` 时透传，wechat-skill 自己 resolve
- `lease_until`：超过此时间 SaaS 应自动 reset 这条为 pending

**Response 200 (no work)**：
```json
{ "rows": [] }
```

Mac 在收到空数组时按 poll interval 退避（1s → 2s → 5s 上限）。

**Response 401 / 403**：bearer 错或 scope 不够 → Mac orchestrate 报错并 sleep 30s 重试。

**Response 5xx**：SaaS 端故障 → Mac 重试。

---

### 3.2 `POST /api/wechat-outbox/:id/done`

**用途**：Mac 已经成功 send 完，回报。

**Request**:
```http
POST /api/wechat-outbox/out_abc123/done
Authorization: Bearer <saas-token>
Content-Type: application/json

{
  "send_message_id": "wechat-msg-server-id-xxx",
  "delivered_verified": true,
  "completed_at": "2026-04-28T10:00:35Z"
}
```

**字段说明**：
- `send_message_id`：wechat-skill 返回的 server_id（可空，老版本不返回）
- `delivered_verified`：wechat-skill 是否在 DB 看到落库（关键 —— 用来 dedupe）
- `completed_at`：完成时间

**Response 200**:
```json
{ "ok": true }
```

**幂等性**：相同 `id` 多次调 done 必须等价（SaaS 用第一次的 send_message_id，忽略后续）。Mac 网络抖动可能重复推。

---

### 3.3 `POST /api/wechat-outbox/:id/fail`

**用途**：Mac send 失败回报。

**Request**:
```http
POST /api/wechat-outbox/out_abc123/fail
Authorization: Bearer <saas-token>
Content-Type: application/json

{
  "error_code": "send_failed | resolve_failed | bridge_unavailable | wechat_blocked | hard_invalid",
  "error_message": "消息发送路径已执行...",
  "retryable": true,
  "failed_at": "2026-04-28T10:00:35Z"
}
```

**`error_code` 枚举**：

| code | retryable | 含义 | SaaS 行为建议 |
|---|---|---|---|
| `send_failed` | true | wechat-skill 返回 send 失败但路径执行了（InputView 冷 / 微信暂时性问题） | 走 backoff 重试 |
| `resolve_failed` | false | 收件人 resolve 不到（wxid 错 / 不在通讯录） | 转 `failed`，通知运营 |
| `bridge_unavailable` | true | wechat-skill REST :18402 不可达（daemon 没起 / Mac 网断） | 走 backoff 重试 |
| `wechat_blocked` | false | WeChat 提示账号风控（这条 send 永久无法成功） | 转 `failed`，紧急通知 |
| `hard_invalid` | false | payload 非法（text 为空 / 超长） | 转 `failed`，标 SaaS bug |
| `lease_expired` | true | Mac 自报 lease 已过期但仍想回报 | SaaS 接受并归档（不应 retry） |

**Response 200**:
```json
{ "ok": true, "next_attempt_at": "2026-04-28T10:02:35Z OR null" }
```

`next_attempt_at` 让 Mac 知道下次什么时候这条会重新 claim 上（仅供日志）。

---

### 3.4 `POST /api/wechat-inbound`

**用途**：Mac 把 wechat-skill SSE 入站事件 push 到 SaaS。

**Request**:
```http
POST /api/wechat-inbound
Authorization: Bearer <saas-token>
Content-Type: application/json
X-Wechat-Signature: t=1730000000,v1=<hex-hmac-sha256>
X-Idempotency-Key: <event_id>

{
  "event_id": "msg-server-id-or-uuid",
  "ts": "2026-04-28T10:00:00Z",

  "normalized": {
    "conversation_id": "wxid_xxx OR group_id@chatroom",
    "is_group": true,
    "sender_wxid": "wxid_caller",
    "sender_display_name": "张三",
    "message_kind": "text | image | voice | video | file | url | quote | recall | system | mp",
    "text": "@bot 你好",
    "is_mentioned": true,
    "mentioned_ids": ["my_bot_wxid"],
    "from_self": false
  },

  "raw_event": {
    "...": "wechat-skill SSE payload v1.10.28 schema 全字段透传"
  }
}
```

**两层字段**：
- `normalized`：稳定子集（最常用 8 个字段，跨版本不变）
- `raw_event`：wechat-skill SSE 原始 payload，**全字段保留**（messageKind / mentionedIds / urlLink / miniProgram / refer / recall / media / ...）。schema 引用 [`wx/schema/sse-payload-v1.10.28.schema.json`](https://github.com/leeguooooo/wechat-skill/blob/main/wx/schema/sse-payload-v1.10.28.schema.json)

订阅者优先用 `normalized`；要做特殊处理（媒体识别 / quote 链 / recall）查 `raw_event`。

**Response 200**:
```json
{ "ok": true }
```

**幂等性**：SaaS **必须** dedupe by `event_id`（或 `X-Idempotency-Key`）。Mac 网络抖动 / SaaS 5xx retry 时同一 event 会重推。

**Response 4xx**：Mac 视为该 event 永久 reject，本地丢弃 + 记 audit log。

**Response 5xx**：Mac retry（指数 backoff，最多 30min）。

## 4. Webhook 签名（防伪 + 防重放）

Mac 推 `/api/wechat-inbound` 时签：

```
sig_payload = "<unix_timestamp>.<request_body_string>"
v1 = HMAC-SHA256(saas-side webhook_secret, sig_payload)
header: X-Wechat-Signature: t=<unix_timestamp>,v1=<hex>
```

`webhook_secret` 是 SaaS 在配置订阅者 token 时同时下发的（独立 secret，不复用 bearer）。

SaaS 端验证：
1. parse `t` + `v1`
2. abs(now - t) > 5 min → reject（防重放）
3. recompute v1 with stored secret → 不 match → reject

参考 Stripe / Slack webhook 签名模式。

## 5. 错误处理 / 重试 / 退化

### Mac orchestrate 行为

| 场景 | Mac 行为 |
|---|---|
| outbox claim 返回 5xx / 网络错 | 指数 backoff（1s → 2s → 5s → 30s），不停重试 |
| outbox claim 返回 401/403 | 立刻 sleep 60s + log 报警（token 错 → 等用户介入） |
| send 调本机 :18402 失败 | 本地短重试 1 次（5 秒后），仍失败 → POST `/done` ❌ → POST `/fail` 带 `bridge_unavailable` |
| webhook POST 5xx | 本地 buffer 最多 100 条，每条最多重试 30min；超出 → 写本地磁盘 audit log + 报警 |
| webhook POST 4xx | 视为 reject，丢 event + 记 audit log |
| lease 过期但 send 已成功 | POST done 加 note "out-of-band"，SaaS dedupe by id |

### SaaS 端必做（spec 强制）

- Outbox claim 是**原子的**（单条 SQL：`UPDATE bot_outbox SET status='claimed', lease_until=? WHERE id IN (SELECT ... LIMIT N AND status='pending') RETURNING *`）
- `done` / `fail` 幂等（同 id 多次调结果等价）
- inbound `event_id` 主键 dedup
- 后台 cron / queue worker 周期性把 `claimed AND lease_until < now` 重置为 `pending`（visibility timeout）

## 7. 跟 v1.11.1 tunnel 的关系

| 用法 | 选哪个 |
|---|---|
| 我有持续 inbound 客户对话 + 异步 send 队列（cherry / 客服 bot / SaaS 集成） | **v1.12 orchestrate** ⭐ |
| 我 CF Worker 偶尔触发，要立刻发条微信，没有 inbound | v1.11.1 tunnel（同步直连，亚秒级） |
| 我同时要两种 | 都装：tunnel + orchestrate 同进程不冲突 |

v1.11.1 tunnel **不会** deprecate，但默认推荐路径是 v1.12 orchestrate。
